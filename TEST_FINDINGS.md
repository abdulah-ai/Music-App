# Starhollow QA findings

## High severity

- [x] **#1 — Locked-out users have no recovery path** | area: LoginScreen | severity: high | The sign-in flow has no forgot-password, reset, or recovery action, so a lost password permanently blocks access from the client. Add a secure recovery flow with clear sent, expired, invalid, and success states. — RESOLVED R1: Added a complete accessible recovery sheet with endpoint-ready request/reset calls and sent, expired, invalid, and success states.
- [x] **#2 — Account security cannot be managed in-app** | area: SettingsScreen / auth flows | severity: high | A signed-in user can view their name and email but cannot change either, change their password, or delete their account. Add authenticated profile, credential, and account-deletion flows with reauthentication and destructive-action safeguards. — RESOLVED R1: Added endpoint-ready profile, password, and permanent deletion controls with current-password reauthentication and typed destructive confirmation.
- [ ] **#3 — Web navigation has no durable URLs** | area: RootNavigator / NavigationContainer | severity: high | The navigator has no linking configuration, so destinations such as Library, Replay, Settings, Player, and Admin cannot be deep-linked, bookmarked, or reliably restored with browser back/forward and refresh. Define stable route URLs and restoration behavior for the web/PWA shell.
- [x] **#4 — Filter failures impersonate an empty library** | area: LibraryScreen | severity: high | Any server-side search or advanced-filter request failure is caught and converted to `[]`, making users believe their music does not exist. Preserve the last result and render a distinct retryable error/offline state instead of a false empty state. — RESOLVED R4: Failed filtered requests now retain the last results and show a distinct retryable error state.
- [x] **#5 — Stale cached music is never identified as stale** | area: libraryStore / HomeScreen / LibraryScreen | severity: high | `libraryStore` records `isStale` after a failed refresh, but no screen reads it, so cached metadata is presented as current without an offline or last-updated warning. Surface the stale state consistently and offer an explicit retry. — RESOLVED R4: Home and Library now share a stale-cache banner with persisted last-updated time and retry.
- [x] **#6 — Library cards contain conflicting nested controls** | area: LibraryMediaView | severity: high | Grid and list cards are whole-row press targets while their “more” buttons are nested press targets, which produces invalid/ambiguous web interaction and risks opening or playing a track when the user intended its menu. Separate the row action and overflow action into sibling controls with independent focus and event handling. — RESOLVED R4: Card actions and 44px overflow buttons are now focusable sibling controls.
- [x] **#7 — Mini-player transport buttons are nested inside navigation** | area: MiniPlayerBar | severity: high | The entire mini-player is a `Pressable` that opens Player, but play, previous, next, and queue buttons are nested inside it. This can cause a transport tap or keyboard activation to also navigate, and creates broken interactive semantics on web; split navigation and transport into sibling targets. — RESOLVED R2: Split track navigation and transport into independently focused sibling targets with isolated handlers.
- [x] **#8 — The waveform is not an accessible scrubber** | area: WaveformScrubber | severity: high | The control is an unlabeled `View` driven only by pointer/touch pan responders, with no slider role, value, keyboard controls, or screen-reader seek actions; its bars are also seeded decoration rather than the track’s waveform. Implement an accessible time slider first, then use analyzed waveform data only as its visual treatment. — RESOLVED R2: Added a semantic time slider with keyboard and assistive seek actions plus an explicitly labeled decorative fallback when analysis is unavailable.
- [x] **#9 — Identify can keep the microphone running after navigation** | area: RecognitionScreen | severity: high | Leaving the Identify tab or unmounting clears timers and subscriptions but never stops an active recorder. Stop and discard capture when focus is lost/unmounted, and make recording state globally visible until cleanup completes. — RESOLVED R3: Blur and unmount now stop, discard, and globally expose capture cleanup until it finishes.
- [x] **#10 — Matched-track download failures can escape the UI** | area: RecognitionScreen / findAndDownload | severity: high | “Add to library” awaits `createDownload` without a `try/catch`, unlike manual search, so a network or API rejection becomes an unhandled promise and leaves the result screen with no recovery. Catch the failure, retain the match, show a retry action, and prevent duplicate submissions. — RESOLVED R3: Guarded imports retain the match and selected source while showing recoverable retry actions.
- [x] **#11 — Recognition imports an unverified first search result** | area: RecognitionScreen / manual search and Add to library | severity: high | Both manual fallback and matched-track import issue a `ytsearch1` download immediately, so the first YouTube result can be the wrong song, remix, live version, or unsafe upload with no preview. Show candidate title, channel, duration, and artwork and require an explicit selection before downloading. — RESOLVED R3: A metadata-rich chooser now requires explicit selection via the marked `/downloads/search` service contract pending server wiring.
- [x] **#12 — Player can render as a completely blank route** | area: PlayerScreen | severity: high | `PlayerScreen` returns `null` when `currentMedia` is missing, even though sidebar and navigation code can still route to Player. Render a branded empty/recovery state that can return to Library or start a queue instead of a blank screen. — RESOLVED R2: Replaced the blank route with a branded night-sky recovery state offering Library navigation and queue startup.
- [x] **#13 — Previous wraps to the last track with repeat off** | area: playerStore / playPrev | severity: high | At the first queue item and before three seconds, Previous always selects the queue’s final item, regardless of repeat mode. Disable Previous or restart the first item when repeat-all is off, and only wrap when the chosen repeat behavior allows it. — RESOLVED R2: Previous now restarts at the queue start with repeat off and wraps only when Repeat All is active.
- [ ] **#14 — Offline users are allowed to start doomed server flows** | area: HomeScreen / RecognitionScreen / global offline behavior | severity: high | Link import and Identify remain fully actionable while the backend is known offline, so users can paste work or record for 15 seconds only to receive a generic failure. Gate server-only actions with an explicit offline explanation while keeping cached browsing and playback available.
- [x] **#15 — One failed admin request blanks the entire console** | area: AdminScreen | severity: high | Users, jobs, logs, feedback, announcements, and stats load in one `Promise.all`; a failure in any endpoint discards every otherwise-usable section. Load each tab independently with per-section errors and retries so operational access degrades gracefully. — RESOLVED R5: Each admin section now loads, fails, retries, and preserves usable data independently.
- [x] **#16 — Admin silently hides records after the first 50** | area: AdminScreen / admin API paging | severity: high | Users, jobs, feedback, and logs request only the default first 50 records and discard each endpoint’s `total`, with no pagination or “showing 50 of N” disclosure. Add server-backed paging or infinite loading and preserve totals in every operational tab. — RESOLVED R5: Every operational tab preserves server totals, discloses the visible count, and can load the next 50 records.
- [x] **#17 — Admin privilege changes are one-tap destructive actions** | area: Admin UsersTab | severity: high | “Make admin” and “Revoke admin” execute immediately without showing the account email, consequences, or a confirmation step. Require an explicit confirmation and reauthentication for role elevation or revocation, then log and visibly acknowledge the result. — RESOLVED R5: Role changes now show the target email and consequences, reauthenticate the acting admin, and visibly acknowledge the logged result.
- [x] **#18 — Admin replies are saved but users cannot read them** | area: Admin FeedbackTab / Settings feedback | severity: high | The console labels the action “Send reply,” but the user-facing app has no inbox, feedback history, or reply display; the backend value is merely stored. Either deliver replies through an in-app feedback thread/notification or relabel this as an internal note so admins are not misled. — RESOLVED R5: Stored admin replies are now consistently labeled as internal admin-only notes that are not sent to users.
- [x] **#19 — Sign-out can erase offline music without warning** | area: SettingsScreen / AppSidebar / AccountPopover | severity: high | Multiple one-tap sign-out actions trigger session cleanup that removes all device offline downloads, yet none confirms this irreversible local deletion. Warn how many downloads will be removed and offer a cancel path before logging out. — RESOLVED R1: Routed every sign-out entry through one cancelable warning that counts the offline downloads removal will clear.
- [ ] **#20 — Global notices compete for the same top lane** | area: AnnouncementBanner / UpdateBanner / Toaster | severity: high | Announcements, updates, and toasts are independently absolutely positioned at the same top inset with nearby z-indexes, so concurrent messages can cover one another and obscure screen controls. Introduce one coordinated notice stack with reserved layout space, priority, queuing, and safe-area handling.

## Medium severity

- [x] **#21 — Sign-in fields do not cooperate with password managers** | area: LoginScreen / RegisterScreen / TextField | severity: medium | Email and password inputs omit appropriate autocomplete and content-type metadata, making autofill, saved credentials, and new-password generation unreliable across web and mobile. Add platform-correct username, current-password, new-password, name, and one-time-code semantics. — RESOLVED R1: Added shared username, current-password, new-password, name, and one-time-code autocomplete and native content-type mappings.
- [x] **#22 — Login failures are not announced accessibly** | area: LoginScreen | severity: medium | The form’s request error is plain styled text with no alert role or live region, so a screen-reader user may receive no indication that submission failed. Move request errors into a shared accessible form-error component and focus or announce it after failure. — RESOLVED R1: Replaced plain request text with a shared assertive alert that receives accessibility focus after failure.
- [x] **#23 — Registration accepts fragile passwords without confirmation** | area: RegisterScreen | severity: medium | Account creation checks only an eight-character minimum and has no confirmation field, typo detection, strength guidance, or compromised-password feedback. Add confirm-password matching and clearer strength/security guidance without blocking password-manager-generated values. — RESOLVED R1: Added password confirmation, email-domain and mismatch typo cues, and nonblocking unique-password strength guidance.
- [ ] **#24 — Paste destroys an existing import draft** | area: HomeScreen / link import | severity: medium | The Paste button replaces the entire URL field, unlike incoming share links which are merged and deduplicated, so carefully prepared multi-link batches can be lost. Merge clipboard links into the draft with duplicate review and provide per-link removal.
- [ ] **#25 — Playlist inspection failures are completely silent** | area: HomeScreen / playlist import | severity: medium | A failed playlist-inspection request is swallowed, leaving users unable to tell whether a link is a single item, an inaccessible playlist, or merely offline. Show a non-blocking inspection error and make the final import scope explicit before submission.
- [ ] **#26 — Today silently retains stale activity data** | area: HomeScreen / In progress widget | severity: medium | Failure to load jobs on focus is ignored, so old progress can remain on screen or active work can disappear without any status or retry. Mark the widget stale/offline and link to Activity’s recoverable error state.
- [ ] **#27 — Global swipe navigation collides with horizontal content** | area: MainTabs / Home / Library | severity: medium | A pan gesture wraps every mobile tab while the screens also contain horizontal shelves and chip scrollers, making deliberate carousel scrolling capable of changing tabs; swiping right from Today unexpectedly opens Settings. Limit paging to an unambiguous edge gesture or remove it in favor of visible navigation.
- [ ] **#28 — Dashboard reset has no undo or confirmation** | area: DashboardCustomizer | severity: medium | “Reset to default layout” immediately overwrites widget order, visibility, density, and accent preferences. Add confirmation or a reversible undo toast, especially because customization changes persist immediately.
- [ ] **#29 — Core appearance settings are hidden in dashboard editing** | area: DashboardCustomizer / SettingsScreen | severity: medium | System/Daylight/Night and accent controls exist only behind “Arrange your hollow,” while Settings contains no appearance section. Put global theme controls in Settings and keep the customizer focused on dashboard layout.
- [x] **#30 — Advanced source filtering omits supported providers** | area: LibraryFilterSheet | severity: medium | The source picker exposes YouTube, Telegram, recognized, and uploaded items but omits TikTok, Instagram, and other URL sources that the same file already knows how to label. Offer every supported source or generate options from actual library/provider data. — RESOLVED R4: The source picker now exposes every source supported by the library query contract.
- [x] **#31 — Filter radio groups expose the wrong accessibility state** | area: LibraryFilterSheet | severity: medium | Choice rows use `accessibilityRole="radio"` but set `selected` instead of `checked`, and the groups themselves have no radiogroup context. Expose checked state, group labels, and clear keyboard traversal for each choice set. — RESOLVED R4: Filter choices now use labeled radiogroups, checked state, and focusable radio controls.
- [x] **#32 — Smart Categories can become an unexplained blank page** | area: SmartCategoriesPane | severity: medium | Empty genre groups are all skipped, but there is no empty state when no track has recognized category metadata. Explain how categories are created and provide a direct “Name tracks” or return-to-All action. — RESOLVED R4: Empty categories now explain genre metadata and offer Name tracks and Return to All actions.
- [x] **#33 — Playlists lack basic organization controls** | area: PlaylistDetailModal / PlaylistsPane | severity: medium | Playlists can be created, played, emptied, or deleted, but cannot be renamed, reordered, given artwork, or have tracks reordered. Add playlist editing and accessible drag/move controls so curated collections can actually be maintained. — RESOLVED R4: Added playlist name, artwork, shelf order, and track order flows against marked API service hooks.
- [x] **#34 — Playlist track removal is a tiny unlabeled target** | area: PlaylistDetailModal | severity: medium | Each playlist row ends in a 17px close icon with only hit slop and no accessibility label, role, or confirmation/undo. Make it a 44px labeled remove button and offer undo to prevent accidental curation loss. — RESOLVED R4: Track rows now use labeled 44px Remove controls with an in-sheet Undo action.
- [x] **#35 — Closing Edit details silently discards work** | area: EditMediaModal | severity: medium | Backdrop tap or system back closes a dirty metadata form without warning, and no dirty state is tracked. Detect changes and confirm discard or preserve the draft until the user explicitly saves or cancels. — RESOLVED R4: Dirty metadata edits now require an explicit discard confirmation on cancel, backdrop, or system back.
- [x] **#36 — Partial bulk deletion exits as if everything succeeded** | area: LibraryScreen / multi-select | severity: medium | Selected tracks are deleted sequentially, but the first failure jumps to `finally` and exits selection mode, obscuring which items were removed and which remain. Continue safely, report per-item outcomes, and retain failed selections for retry. — RESOLVED R4: Bulk delete now continues per item and retains failed selections with an outcome summary.
- [x] **#37 — Bulk native download can launch dozens of external intents** | area: LibraryScreen / Download all | severity: medium | On platforms without the web cache path, “Download all” sequentially opens up to 50 URLs and tells the user to repeat for the rest, creating an overwhelming OS-level interaction instead of managed offline storage. Use a native download manager with a cancellable queue, storage checks, and consolidated progress. — RESOLVED R4: Native downloads now use a cancellable one-at-a-time managed handoff queue with progress and a marked storage-manager hook.
- [x] **#38 — Queue editing stops at remove-only** | area: QueueList / playerStore | severity: medium | Users cannot reorder upcoming tracks, clear the queue, save it as a playlist, or undo a removal. Add accessible move/reorder actions and explicit queue-level controls while protecting the current item. — RESOLVED R2: Added accessible move controls, current-safe clearing, playlist saving, and announced removal undo.
- [x] **#39 — Queue rows repeat the nested-control problem** | area: QueueList | severity: medium | A whole queue row is pressable to jump while its remove button is nested inside that row, leading to ambiguous focus and click propagation on web. Render jump and remove as sibling actions with a clear current-track non-action state. — RESOLVED R2: Queue jump and edit actions are now siblings while the current track is an explicit non-action summary.
- [x] **#40 — Mini-player queue preview is a dead-end subset** | area: MiniPlayerBar / QueuePreview | severity: medium | The preview shows only the next five items and offers neither remove/reorder controls nor a route to the complete queue. Add an “Open full queue” action and make the preview’s limited scope explicit. — RESOLVED R2: The preview now states its five-track scope and opens the full queue directly in Player.
- [x] **#41 — Player artwork cannot be explored** | area: PlayerScreen / Artwork | severity: medium | The dominant artwork is purely static: users cannot tap to inspect it full-screen, view the uncropped image, or see a useful high-resolution/loading failure state. Add a focused artwork viewer that preserves the night-sky presentation and handles missing/low-resolution art gracefully. — RESOLVED R2: Added a focused full-screen night-sky viewer with uncropped best-available art and graceful fallback status.
- [x] **#42 — Mobile playback has no volume or mute control** | area: PlayerScreen | severity: medium | Volume and mute controls are rendered only on desktop, leaving mobile web and tablet users without in-app volume adjustment. Provide a compact accessible volume control on mobile while respecting hardware-volume conventions on native devices. — RESOLVED R2: Added compact labeled mute and volume controls on mobile web and tablet layouts while leaving native hardware volume authoritative.
- [x] **#43 — Unavailable transport actions still look active** | area: PlayerScreen / playerStore | severity: medium | Previous and Next remain enabled even when there is no valid destination, and reaching the queue end gives no explanation. Compute availability, expose disabled state and hints, and keep repeat/shuffle behavior predictable. — RESOLVED R2: Centralized transport availability, disabled invalid skips with hints, and removed repeat-off manual wrapping at queue end.
- [ ] **#44 — Lyrics errors look identical to missing lyrics** | area: LyricsView | severity: medium | `fetchLyrics` has no catch/error state, so network, provider, authentication, and parsing failures all collapse into “No lyrics found.” Preserve the failure reason, keep cached lyrics when possible, and provide retry.
- [ ] **#45 — Lyrics auto-scroll ignores reduced-motion preference** | area: LyricsView | severity: medium | Every active-line change calls animated scrolling even when reduced motion is enabled. Use immediate or user-controlled positioning under reduced motion and pause auto-follow after manual scrolling.
- [ ] **#46 — Synced lyrics are poorly exposed to assistive technology** | area: LyricsView | severity: medium | Tappable lyric lines have no button role, seek-time label, selected/current state, or live indication of the active line. Add accessible seek actions and a restrained current-line announcement strategy that does not read every line over the music.
- [x] **#47 — Recording has no cancel-and-discard action** | area: RecognitionScreen | severity: medium | Once listening starts, the only visible action stops and submits the clip for recognition; users cannot cancel a bad recording without sending it. Add a clearly separate Cancel action that stops, deletes, and never uploads the clip. — RESOLVED R3: Listening now presents a separate cancel action that stops and deletes without recognition upload.
- [x] **#48 — Capability outages are misreported as missing configuration** | area: RecognitionScreen | severity: medium | If the capabilities request fails, the UI hardcodes humming as unavailable and tells users ACRCloud is not connected, even when the real problem is network or server failure. Distinguish loading, configuration, and connectivity errors and offer retry. — RESOLVED R3: Capability loading, connectivity failure, and confirmed server configuration now have distinct retryable states.
- [x] **#49 — Recent scans are non-actionable and arbitrarily truncated** | area: RecognitionScreen / scanHistoryStore | severity: medium | Identify shows only four of up to ten stored scans, with no View all, rescan, search, or add-matched-track action from history. Provide a complete history surface with useful row actions and clear retention behavior. — RESOLVED R3: View all now exposes every retained scan with search, rescan, matched-track add, clear, and retention guidance.
- [ ] **#50 — “Hide finished” does not actually remember the choice** | area: JobsScreen | severity: medium | The action removes finished jobs only from local component state, so every focus or reload fetches them back. Persist the visibility filter or rename it to a temporary dismissal and explain how to restore hidden entries.
- [ ] **#51 — Activity history neither scales nor leads anywhere** | area: JobsScreen | severity: medium | All returned jobs are mapped inside a `ScrollView`, and completed rows cannot open their media, source, details, or Player. Use a virtualized/paginated list and make successful and failed rows actionable detail destinations.
- [ ] **#52 — Activity cancellation is too easy to trigger** | area: JobsScreen / HomeScreen | severity: medium | A single close-icon tap cancels an active import with no confirmation, even for long playlist or Telegram jobs. Require confirmation that names the job and preserve a retry/restart path after cancellation.

## Low severity

- [ ] **#53 — Replay can list ghost artists from deleted tracks** | area: ReplayScreen | severity: low | Top tracks are filtered against the current library, but top artists are calculated from all history without the same filter, so deleted media can leave unexplained artist rankings. Reconcile both lists against current media or visibly label historical/deleted sources.
- [ ] **#54 — Replay is a fixed snapshot rather than an exploratory recap** | area: ReplayScreen | severity: low | The view is locked to 30 days and offers no alternate window, comparison, play-all queue, or export/share action. Add a few meaningful time ranges and listening actions without turning the calm recap into a dense analytics dashboard.
- [ ] **#55 — Telegram status can remain “Checking…” forever** | area: SettingsScreen / Connection | severity: low | A failed Telegram status request sets the value back to `null`, which the status row always renders as pending instead of failed. Show a retryable unavailable state and distinguish it from a genuinely in-progress check.
- [ ] **#56 — Library export is not a complete portable backup** | area: SettingsScreen / Export library | severity: low | JSON export omits playlists, favorites, pins, listening history, recognition history, custom metadata state, and offline manifest, while native merely copies a potentially huge payload to the clipboard. Offer a versioned backup file/share flow with explicit included data and a tested import/restore path.
- [x] **#57 — Toasts disappear without accessible control** | area: Toaster / toastStore | severity: low | Toasts have no alert/live-region semantics, cannot be dismissed, and vanish after 3.2 seconds even when they contain important failure text. Announce by tone, allow dismissal or pause, and keep actionable errors available long enough to understand. — RESOLVED R1: Added tone-aware live semantics and dismiss controls, with error toasts persisting until dismissed and longer informational timing.
- [x] **#58 — Glass sheets do not manage keyboard focus** | area: CompactGlassSheet / library and player overlays | severity: low | The shared sheet marks itself modal but does not move focus into the panel, trap web keyboard focus, support Escape explicitly, or restore focus to the opener. Add consistent focus lifecycle and dialog semantics across every overlay. — RESOLVED R1: Added dialog semantics, initial focus, web focus trapping and Escape handling, native accessibility focus, and opener restoration to the shared sheet.
- [x] **#59 — Editing an admin email saves unexpectedly on blur** | area: Admin UsersTab | severity: low | Leaving the field immediately submits the draft, then closes edit mode even after an error, making accidental or malformed changes easy. Provide explicit Save/Cancel controls, validate email shape, and retain failed drafts. — RESOLVED R5: Email editing now has explicit Save and Cancel actions, validates format, and retains failed drafts with inline errors.
- [x] **#60 — Admin data has no search or operational filters** | area: Admin Users / Jobs / Feedback / Logs tabs | severity: low | The API supports job status, feedback status, and event-type filters, but the console exposes none and offers no user/job search or sort controls. Add server-backed filtering and search so admins can isolate failures, accounts, and unresolved work quickly. — RESOLVED R5: All operational tabs now issue scoped filter, search, and sort requests, with search and sort added as clearly marked endpoint-ready service parameters.
